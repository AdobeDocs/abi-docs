# Core Concepts

## What Brand Intelligence does

Adobe Brand Intelligence (ABI) validates creative assets against your organization's brand guidelines. You submit assets - images, documents, layouts - and ABI checks each one against a set of brand rules (colour palette, typography, logo usage, and so on). For each asset, ABI returns a list of violations with severity levels and human-readable messages.

Validation runs against **organization-level brand guidelines** and optionally against **campaign-specific guidelines** when a `campaignId` is provided.


## Async job model

Validation is asynchronous. You submit a batch of assets and receive a `jobId` immediately; ABI processes each asset in the background and you poll for results.

```
POST /api/v1/review-and-approve      → 202 Accepted  { jobId }
GET  /api/v1/review-and-approve/{jobId}   → job status + counts
GET  /api/v1/review-and-approve/{jobId}/items → per-asset results
```

This design keeps the API responsive for large batches. A single job can contain up to 100 assets.


## Resource hierarchy

```
Job
└── Items (one per submitted entry)
    ├── Assets (one per image; one per page for multi-page documents)
    └── Comments (reviewer feedback on the item)
```

**Job** - represents a single validation run. Created by `POST /api/v1/review-and-approve`. Tracks overall status and aggregate counts (succeeded, failed, pending).

**Item** - represents one submitted entry within a job. A single item can produce multiple assets — for example, a PDF document generates one asset per page. Contains a `validationSummary` and a `validationErrors` list aggregated across all its assets.

**Comment** - structured reviewer feedback attached to an item. Comments can be created by your application (`errorSource: user`) or generated by the validation pipeline (`errorSource: pipeline`, read-only).


## Job lifecycle

A job moves through the following states:

| Status | Meaning |
|--------|---------|
| `not_started` | Job accepted, processing has not begun. |
| `running` | One or more items are being validated. |
| `succeeded` | All items completed successfully. |
| `partially_succeeded` | Some items completed; others failed. |
| `failed` | All items failed to validate. |
| `canceling` | Cancellation requested; in-flight items are finishing. |
| `canceled` | Job was canceled before completion. |

Once a job reaches `succeeded`, `partially_succeeded`, `failed`, or `canceled` it will not change state again. Poll until you see one of these terminal states before processing results.


## Item status values

Each item within a job moves through its own status independently:

| Status | Meaning |
|--------|---------|
| `NOT_STARTED` | Item queued, processing has not begun. |
| `WAITING` | Item is waiting for a processing slot. |
| `PROCESSING` | Asset is being downloaded and prepared. |
| `POLLING` | Validation is in progress. |
| `RUNNING` | Validation is actively running. |
| `SUCCEEDED` | Validation completed. Results are available. |
| `FAILED` | Validation failed. See `error` for details. |


## Validation results

Each item in the results contains:

- **`validationSummary`** - a `summaryText` string with a human-readable narrative of the validation outcome.
- **`validationErrors`** - list of individual violations. Each violation includes:
  - `errorId` - unique identifier for this violation instance.
  - `errorSummary` - human-readable description of the violation.
  - `assetAttributions` - regions within the submitted asset that triggered the violation. Each entry includes a `documentUrl` (signed URL to the asset), `chunkUrl` (signed URL to the specific region), and `regions` (bounding box in normalised 0–1 coordinates).
  - `corpusAttributions` - chunks from the brand guidelines corpus that the violation was assessed against. Each entry includes a `documentId`, `documentUrl`, and `chunkUrl` pointing to the guideline source.

An item with an empty `validationErrors` array is fully compliant.

See the [API Reference](../../api/index.md) for the full `BrandValidationError` schema including reviewer feedback fields.


## Pagination

The jobs list endpoint supports cursor-based pagination. Pass `pageSize` (1–100) and `cursor` query parameters. The response includes a `nextCursor` field; when it is `null`, you have reached the last page.


## Authentication

Every request requires an IMS Bearer token in the `Authorization` header. Tokens are valid for 24 hours. See [Authentication](../authentication/index.md) for setup instructions.
